# Everline Agent 約束

本文件適用於整個 Everline Design 倉庫。未來 coding agent、design agent、文件 agent 與自動化工具在新增或修改內容時，都必須遵守以下規則。

## 專案定位

- Everline 是 Taylor Kanban 與未來延伸產品共用的圖形介面設計語言與設計系統基礎。
- 本倉庫目前以規格、設計研究、向量原始檔與視覺資產管理為主，不是可執行應用程式，也不是已完成的前端元件庫。
- 工作目標是讓設計原則、token、元件、圖示、品牌資產與匯出結果可規範、可追蹤且可持續維護，而不是只保存單次繪圖成果。
- `docs/spec.md` 是主要規格與事實來源。執行工作前，先讀 `docs/HANDOFF.md` 快速掌握現況，再依任務需要閱讀 `README.md`、`docs/STATUS.md`、`docs/NEXT_ACTION.md` 與 `references/gui-components-reference.md`。

## 指示與事實優先順序

發生衝突時依下列順序處理：

1. 使用者在目前任務中的明確要求。
2. 本文件與更接近修改範圍的 `AGENTS.md`。
3. `docs/spec.md` 中的 FROZEN Decisions、Contracts 與 Invariants。
4. 已驗證的倉庫內容、狀態文件與測試結果。
5. 研究參考、建議結構與尚未確認的規劃。

不得把建議、待辦、參考案例或推測寫成已完成事實。若需求會改變專案定位、主要資料夾責任或 Everline 與 Taylor Kanban 的關係，先停止實作並取得使用者確認。

## 工作開始前

- 先讀取與任務直接相關的文件、資產、引用與 Git 狀態；不要猜測可查證內容。
- 先說明本次成功條件、會修改的檔案類型及預期產物。
- 確認工作屬於設計原則、design token、元件規格、圖示／圖形資產、匯出規則或專案文件之一。
- 若新增 token、component、icon 或品牌資產，先確認其產品情境、語意、重複使用價值與來源。
- 若無法判斷內容應放置的位置或正式程度，保留為待確認項目；不要自行建立新的大型分類或抽象層。

## 凍結邊界

- Everline 必須維持為「圖形介面設計語言與設計系統專案」，不得被改寫為單一應用程式、單一品牌圖檔或一次性繪圖任務。
- `docs/spec.md` 必須維持為主要規格入口。
- `works/` 必須保存可編輯的設計來源（Illustrator 向量原始檔、`works/html/` 下的 HTML/CSS 候選元件原型），不得作為一般匯出圖片目錄。
- `exports/` 必須保存由向量來源產生的圖片或可交付視覺資產，不得成為唯一設計來源。
- `references/` 只保存研究與參考素材，不得被視為正式可用的 Everline 資產來源。
- `tokens/` 只在有產品或工具需要消費設計值時建立和使用，不得放入一般圖片或向量原始檔。
- Taylor Kanban 倉庫尚未驗證前，不得聲稱其框架、路由、元件結構、資料模型、設計實作或其他內部細節。
- 初版不直接實作前端元件庫，也不重構 Taylor Kanban。若需求進入實作階段，先建立獨立技術規格並確認目標技術棧。
- 不得僅因未來可能需要而建立大型空目錄、空 schema 或未被需求支持的流程。

## 資料夾責任

| 路徑 | 責任 | 禁止事項 |
| --- | --- | --- |
| `docs/` | 規格、決策、使用指南、狀態與設計系統文件 | 不混入一般匯出圖或可執行實作 |
| `works/` | 可編輯設計來源：Illustrator `.ai`（構思與圖示，`works/illustrator/`）、HTML/CSS 候選元件原型（`works/html/`） | 不存放一般匯出圖片；不任意搬移既有來源 |
| `exports/` | 從 `works/` 產生的 PNG、SVG、WebP 等交付物 | 不作為唯一來源；不可收錄來源不明的正式資產 |
| `references/` | 產品截圖、風格研究、競品與外部參考 | 不當成正式成品；不移除來源與授權註記 |
| `tokens/` | JSON、CSS variables 等可被產品或工具消費的設計值 | 不存放圖片、原始向量檔或純研究筆記 |

只有在有實際內容時才建立尚不存在的 `exports/` 或 `tokens/`。新增其他頂層目錄前，必須說明其責任、內容類型及與既有目錄無法合併的理由。

## 設計系統建構規則

### 設計原則

- 規則必須描述可重複使用的視覺或互動判斷，不得只描述單一畫面外觀。
- 應逐步涵蓋整體氣質、色彩、排版、間距、圖示、線條、形狀、插圖、元件狀態與互動回饋。
- 參考 Material、Fluent、Carbon、Atlassian、Apple 或 Figma 時，只能提煉行為與結構；不得直接複製其品牌樣式作為 Everline 成品。

### Design token

- 建立 token 前先定義語意、用途、作用範圍與消費者，不要只把目前畫面中的數值機械轉錄。
- 優先使用語意名稱，並清楚區分全域基礎值、語意 token 與元件專用 token。
- 選擇 JSON、CSS variables 或其他格式前，必須先驗證實際整合需求；不得預設前端框架。
- token 變更應說明對既有元件、模式、可讀性與相容性的影響。

### 元件規格

- 每個正式元件至少定義用途、適用與不適用情境、結構、狀態、變體、互動、內容規則及無障礙考量。
- 狀態命名需一致且有明確語意；目前候選包含 `default`、`disabled`、`danger`、`outline`、`focused`、`on` 與 `off`，正式採用前仍需由元件規格確認。
- 不得以一張設計草圖直接宣告元件已定稿；應區分候選、草稿、已審核與穩定狀態。
- 新元件必須證明有跨畫面或跨產品的重用價值；一次性組合應留在產品層，不要過早升格為系統元件。

### HTML/CSS 候選元件原型（2026-07-19 新增，同日修正流程順序）

- Illustrator 仍是構思與草稿階段的入口（使用者慣用工具），保留原有角色；圖示仍以既有規則為準，維持 SVG。
- SVG 候選＋人類審查階段不被取代：元件大致樣貌在 Illustrator 定案後，先以 SVG 產生候選稿供人類視覺審查。第一批（既有 18 類）留在 `works/everline-components-master.svg`；第二批起改用獨立的批次檔案（例如 `works/everline-components-batch2.svg`），不再併入同一份總表，避免單一檔案過度膨脹、增加後續 agent 讀取負擔。
- SVG 候選稿經人類確認沒有問題後，才轉為 `works/html/` 下的 HTML/CSS：真實文字、flexbox／CSS custom properties 版面，以及 `:hover`／`:focus`／`:disabled` 等原生互動狀態，取代手繪每個狀態的做法。
- 轉為 HTML/CSS 屬於「畢業」：該元件之後以 `works/html/` 的 HTML/CSS 版本為候選規格權威來源，對應 SVG 版本標記為歷史快照（比照既有 `superseded-by` 慣例），不再更新。
- CSS custom properties 必須對應 `tokens/` 下既有語意 token 命名，不得另建一套命名系統。
- 畢業前必須先做 token 完整性核對：列出該批 SVG 實際使用的每個尺寸與顏色數值，逐一確認 `tokens/` 裡有對應項目、且數值與 SVG 一致；查無對應項目或數值對不上時，先修正或補上 token，再動 HTML/CSS，不可先寫 CSS、事後才回頭補 token。凡 `tokens/` 裡標記 `candidate`／`unverified` 的值，畢業時一律以已審查通過的 SVG 實際繪製結果重新核對取代，不得直接當作正確值原封不動沿用（已審查通過的內容優先於未確認候選值，見「指示與事實優先順序」）。
- 畢業完成後的驗證必須包含像素等級量測（關鍵尺寸量測、關鍵色塊 RGB 採樣），與審查通過的 SVG／PNG 逐項比對數值；只用肉眼比對截圖不算完成驗證，因為無法可靠抓出個位數百分比等級的尺寸落差，或背景色與前景元素完全同色導致視覺融合的狀況。
- 徽章、計數、狀態文字等「內容應與畫面元素數量一致」的地方，須獨立檢查邏輯一致性；不能假設 SVG 審查通過就代表內容本身沒有問題，SVG 階段的疏漏也可能被原樣帶進 HTML/CSS。
- 這仍是候選／草稿階段，不代表進入前端元件庫實作；技術棧未驗證前，候選稿須維持框架中立的原生 HTML/CSS/JS，不得預設 React、Vue 等特定前端框架。
- 既有 `works/everline-components-master.svg`（第一批 18 類）不強制立即轉為 HTML/CSS；個別元件確定要轉換時才動作。

### 圖示、字型與品牌資產

- 新增第三方素材前，確認來源、版本、授權、允許用途與必要歸屬聲明。
- Lucide 與 `references/` 內容保留其上游權利；根目錄 Apache-2.0 不會改變第三方素材的原始授權。
- 圖示應另行定義 Everline 的尺寸、線寬、視覺重量、對齊、狀態與命名規則，不得因使用現有圖示庫就省略系統規範。
- 第三方圖示庫只作原型或參考；可商業化或獨立授權的正式核心資產應重視原創性、來源追蹤與權利邊界，不得整套直接依賴外部圖示。
- 商標、產品截圖與參考圖片不得被重新包裝為 Everline 自有資產。

## 資產與匯出規則

- 新資產檔名使用小寫英文、數字與連字號，名稱應包含用途或主題；避免空格、模糊縮寫、自動編號與只有版本號的名稱。
- 同一設計的來源與匯出檔應保留相同主體名稱，並以格式、尺寸、模式或用途尾綴區分。
- 從向量檔匯出前，必須知道來源檔、輸出格式、尺寸或密度、使用情境與目標產品。
- 匯出檔必須放在 `exports/` 或其合適子目錄，並透過檔名、相鄰說明或資產索引維持來源可追溯性。
- 無法追溯向量來源的圖片只能放在 `references/` 或標記為待整理，不得視為正式設計系統資產。
- 同一資產適用多個產品時，優先使用通用位置與命名，不要複製多份造成版本飄移。
- 不以匯出檔覆蓋向量來源，也不因建立新版就刪除仍可能被引用的舊來源。
- `works/illustrator/everline_p1.ai`（2026-07-19 經使用者同意由 `works/everline_p1.ai` 搬入 `works/illustrator/`）是既有來源；再次重命名、搬移或刪除前必須先確認外部引用、匯出依賴及回復方式，並取得使用者同意。

## 文件規則

- 修改 `docs/spec.md` 前必須完整閱讀該文件，並指出修改屬於補充、澄清或修正。
- 規格修改後必須保留專案概述、核心問題、目標、非目標、功能清單、資料夾結構、技術與工作流考量、成功指標及完整工程防護層。
- 重大設計方向、資料夾責任或公開介面變更應記錄原因、影響、遷移方式與未決風險。
- 文件必須明確區分：已確認事實、設計決策、候選方案、待辦與未決問題。
- 外部研究需保留來源連結、用途及授權注意事項；未重新驗證時不要宣稱來源內容仍是最新版本。
- 中文與其他非 ASCII 內容一律以 UTF-8 讀寫；發現亂碼時先檢查編碼，不得直接覆寫檔案。

## 變更安全

- 修改前檢查 Git status 與相關引用；保留使用者或其他 agent 的未提交變更。
- 不處理任務範圍外的重構、重新命名、格式化或資產清理。
- 未經明確要求，不自行 commit、push、建立 PR、合併、發布或部署。
- 刪除、覆蓋、搬移設計來源，或執行其他不可逆操作前，必須說明影響並取得使用者明確同意。
- 不提交憑證、token、私人資料、未確認可公開的產品內容或來源不明的商用資產。

## 驗證要求

依變更類型執行相稱的驗證：

- 文件：檢查必要章節、相對連結、路徑、UTF-8 編碼、Markdown 結構、待辦狀態及與 `docs/spec.md` 的一致性。
- 資產：確認檔案可開啟、格式與尺寸正確、命名合規、來源可追溯、授權已記錄。
- 匯出：將輸出與向量來源視覺比對，確認尺寸、裁切、透明度、色彩模式、密度及用途。
- Token：驗證格式可解析、名稱唯一、引用存在、模式切換完整，並檢查破壞性變更。
- 元件規格：確認用途、狀態、變體、互動、內容與無障礙要求完整，且沒有把研究參考寫成既有產品事實。
- 前端實作：先依獨立技術規格執行專案既有的測試、型別檢查、lint、build 與必要的視覺回歸驗證。

任何驗證未執行或無法執行時，必須明確回報原因與剩餘風險，不得宣稱已通過。

## 完成條件

只有同時符合以下條件才可宣告完成：

- 需求與相關驗收條件已完整處理。
- 新內容位於正確責任邊界，且未破壞 FROZEN Decisions 或 Invariants。
- 來源、命名、引用、授權與可追溯性已依變更類型確認。
- 文件與實際倉庫狀態一致，沒有把規劃、研究或未驗證資訊寫成事實。
- 已執行相稱驗證並回報結果；未驗證風險已清楚列出。
